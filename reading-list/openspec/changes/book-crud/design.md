## Context

Greenfield feature. No existing book management exists. The project uses Spring Boot 3.5.13 with Spring Data JPA backed by H2 in-memory storage. Thymeleaf handles server-rendered UI.

## Goals / Non-Goals

**Goals:**
- Standard layered architecture: Controller → Service → Repository
- Single `Book` entity persisted via JPA
- REST API returning JSON
- Thymeleaf UI for listing and managing books
- Input validation via Bean Validation annotations

**Non-Goals:**
- Authentication or authorization
- Pagination (deferred to a future spec)
- External database (H2 in-memory is sufficient for now)
- Client-side JavaScript frameworks

## Decisions

**UUID as primary key** — avoids sequential ID leakage, natural fit for REST resource identifiers. Generated by the application layer, not the database.

**Status as enum** — `UNREAD`, `READING`, `FINISHED` are a closed set. Using a Java enum mapped as `@Enumerated(EnumType.STRING)` keeps DB values human-readable and safe from ordinal drift.

**Service layer owns validation** — controller handles HTTP concerns only; service enforces business rules. This keeps controllers thin and services testable without MockMvc.

**`addedAt` set server-side** — set at creation time in the service, never accepted from the client. Prevents client clock skew.

**DTO pattern** — `BookRequest` for inbound data (validated), `Book` entity never exposed directly. Keeps API contract decoupled from persistence model.

## Risks / Trade-offs

- H2 in-memory → data lost on restart. Acceptable for a pet project; migration to persistent DB is straightforward (swap datasource config).
- No pagination → list endpoint returns all books. Acceptable at small scale; flagged for future improvement.
